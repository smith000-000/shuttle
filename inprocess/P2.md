# P2: Secondary Runtime Integration

## Purpose
Track the active P2 mega-epic as the architecture and integration pass for non-builtin runtimes, with `codex_sdk` as the first authoritative secondary runtime.

## Current Decision Summary
- `inprocess/P2.md` is the active mega-epic tracker.
- P1 is retired and archived as `completed/p1-hardening-and-runtime-expansion.md`.
- Shuttle remains the sole owner of tmux interaction, shell observation, approvals, transcript mutation, patch application, and task/session persistence.
- P2 deepens runtime behavior behind `internal/agentruntime`; it does not introduce a plugin marketplace or alternate controller path.
- Runtime adapters must continue to target the controller-owned `agentruntime.Host` contract.
- Runtime selection is session-authoritative for agent reasoning. Shuttle still constructs each turn, but it must send every agent decision turn for that task/session back to the selected runtime. P2 continues this full outsource model and does not introduce dual-runtime delegation.
- Host callbacks are not fallback. `Respond`, `InspectContext`, `SynthesizeStructuredEdit`, and `ValidatePatch` remain Shuttle-owned services used by the selected runtime.
- Silent per-turn builtin fallback is not allowed for an authoritative external runtime. If the selected runtime cannot remain the reasoning owner, Shuttle stops and requires an explicit retry or runtime switch.
- `codex_sdk` is the first non-builtin authoritative runtime target in P2. The current phase validates Shuttle turn handling and operator UX through the local `codex` CLI bridge while also carrying a `codex_app_server` bridge runtime for real app-server turn handling with a long-lived app-server process and in-memory per-task native thread reuse.
- `pi` is detectable but rejected for authoritative selection until it supports the full required request-kind set.
- `auto` should prefer the best installed full-parity authoritative runtime, then fall back to `builtin` when none qualify.
- Runtime selection is no longer startup-only UX: Shuttle now exposes it through `F10 -> Runtime` and persists the selected runtime similarly to saved provider/model state, while still letting explicit runtime flags win for that launch.

## Authority Boundary
Operations a runtime may request only through `agentruntime.Host`:
- provider/model response generation through `Host.Respond`
- authoritative shell-context refresh through `Host.InspectContext`
- controller-owned structured-edit synthesis through `Host.SynthesizeStructuredEdit`
- controller-owned patch validation through `Host.ValidatePatch`

Operations that remain impossible for a runtime to own directly:
- tmux pane control or shell injection
- shell reads/writes outside host callbacks
- approval gating decisions
- patch application
- transcript mutation
- task/session persistence
- command execution lifecycle ownership

The current `Host` interface is sufficient for the first P2 implementation slice.

## Required Parity Matrix
The following request kinds are required before a secondary runtime can be considered authoritative:
- `user_turn`
- `proposal_refinement`
- `approval_refinement`
- `continue_plan`
- `continue_after_command`
- `continue_after_patch_apply`
- `compact_task`
- `execution_checkin`
- `lost_execution_recovery`
- `resume_after_take_control`

Parity means compatible Shuttle-side behavior for:
- `message`, `plan`, `plan_step_statuses`, `proposal`, and `approval`
- inspect-context recursion
- structured-edit synthesis
- invalid-patch repair and validation interception
- controller-owned auto-run/auto-apply policy

Metadata differences are acceptable only for runtime provenance and authority/debug details.

## Failure Policy
- Authoritative runtime failure means launch failure, crash, unusable/contract-invalid output after retries, or an explicit user runtime switch.
- On authoritative runtime failure, Shuttle should preserve controller state, emit a visible runtime failure/error, and stop further agent continuation.
- Shuttle should not auto-switch to builtin or silently mix builtin reasoning into the same session.

## Recommended Implementation Order
1. Freeze the authority contract and explicit request-kind emission.
2. Keep runtime metadata and UX aligned with authoritative selection.
3. Implement and verify `codex_sdk` first through the local `codex` CLI executable so Shuttle can validate request kinds, approvals, transcript behavior, and operator UX without taking on session-thread management immediately.
4. Continue validating the turn contract through both the CLI-backed `codex_sdk` bridge and the persisted-thread `codex_app_server` bridge, then harden app-server recovery/compaction behavior before letting `auto` prefer it.
5. Hold `pi` behind explicit rejection until full parity exists.

## Acceptance Gates
- Non-builtin runtimes must remain behind `internal/agentruntime`.
- Shuttle must remain the only authority for shell state, approvals, patch apply, transcript mutation, and task/session persistence.
- An authoritative external runtime must own the full required request-kind set for the session.
- No silent builtin fallback may handle ordinary continuation turns for an authoritative external runtime.
- `go test ./internal/... -count=1` must remain green throughout P2 work.

## Deferred / Out of Scope
- No dual-runtime delegation or builtin-driven escalation here; that work is tracked in [P5](P5.md).
- No generalized runtime marketplace in P2.
- No alternate controller path owned by `pi`, `codex_sdk`, or another framework.
- No release-packaging hardening here; that moves to P3.
- No memory/auto-compaction/rules work here; that moves to P4.
