# P4: Memory, Auto-Compaction, And Rules

## Purpose
Capture the future work for durable memory, automatic compaction, and rule handling across the Shuttle controller/runtime workflow.

## Initial Scope
- Durable task and session memory that remains Shuttle-owned, inspectable, and auditable
- Auto-compaction triggers, retention rules, and recovery behavior
- User rules, project rules, and runtime/model rules with explicit precedence
- Integration of memory and rules into the existing controller/runtime contract without bypassing Shuttle state authority

## Guiding Constraints
- Shuttle remains the source of truth for stored memory, compaction state, and applied rule context
- Memory and rules must be inspectable and debuggable from the product side
- Auto-compaction should improve context management without making task state opaque or surprising
- Runtime adapters may consume normalized memory/rule context, but they should not own persistence or precedence policy

## Notes
- P4 is planned work, not the active mega-epic.
- P2 remains the active epic until explicitly retired.
- This epic replaces the earlier placeholder P4 scope.
