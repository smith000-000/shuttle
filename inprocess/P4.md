# P4: Memory, Context Compaction, and Agent Rules

## Purpose
Track the P4 mega-epic for durable memory, automatic context-pressure handling, configurable pruning, and explicit rule management so Shuttle can stay reliable over long-running host-specific workflows.

## Current Decision Summary
- `inprocess/P4.md` is the long-lived tracker for memory and context-governance work.
- Memory should be host-aware and confidence-scored rather than a single global notes bucket.
- Auto-compaction should activate before provider hard limits are reached and should preserve operator-critical shell facts.
- Memory retention and pruning must be configurable with explicit operator controls and transparent defaults.
- Rules should be first-class artifacts (not ad hoc prompt text) with precedence and auditability.

## Problem Statements from Intake
1. Operators need recall across sessions for host-specific work (example: “Where was that SnapRAID error we fixed a while back?”).
2. Context currently depends too heavily on manual `/compact`, especially when large shell output risks overrunning model context windows.
3. Memory accumulation needs lifecycle management (retention, pruning, and cleanup controls).
4. Operators need a durable “rule” surface for persistent guidance the agent should always respect.

## Epic Workstreams

### 1. Host-Scoped Memory
- [ ] Define the memory unit model (fact, event, remediation, decision, and reference).
- [ ] Add host/session attribution fields (host identity, cwd/project hints, timestamp, confidence, and source pointers).
- [ ] Add retrieval primitives for “when/where did we fix X?” style lookups.
- [ ] Prevent unsafe memory reuse across mismatched hosts or stale confidence.
- [ ] Expose memory citations back to the transcript so operators can inspect provenance.

### 2. Automatic Context Compaction
- [ ] Add provider-capability-aware context budget estimation before each model turn.
- [ ] Trigger preflight auto-compaction when estimated token usage crosses configurable thresholds.
- [ ] Prefer structured shell summarization that preserves actionable artifacts (paths, commands, errors, diffs).
- [ ] Keep `/compact` as a manual override while recording when auto-compaction was applied.
- [ ] Add regression coverage for long-output command sessions and repeated compact/retry loops.

### 3. Memory Retention and Pruning Management
- [ ] Add memory retention policy config (age/size/importance thresholds).
- [ ] Add manual prune controls (target host, task range, tag/category).
- [ ] Add startup/background pruning jobs with conservative safety rails.
- [ ] Show memory storage/health status in settings or inspect flows.
- [ ] Document privacy implications and operator controls for stored memory.

### 4. Rule System
- [ ] Define a rules data model with scopes (global, workspace, host, task).
- [ ] Define precedence between system/runtime constraints, operator rules, and transient task instructions.
- [ ] Add CRUD UX for rules in the TUI/settings flow.
- [ ] Inject active rules into prompt construction with concise formatting and traceability.
- [ ] Add conflict handling UX when rules collide with current task requests.

## Candidate Architecture Direction
- Controller-owned memory/rule services under `internal/controller` with persistence backing.
- Provider-facing prompt assembly consumes normalized “memory snippets” and “active rules” rather than raw storage rows.
- TUI surfaces inspection and management actions while controller remains source-of-truth for eligibility and precedence.
- Add provider-aware compaction policy hooks near existing context assembly and usage accounting paths.

## Open Questions
- Should host identity be keyed primarily by prompt-detected hostnames, explicit probe metadata, or both?
- What minimum provenance is required before a memory fact is considered reusable?
- How aggressively should auto-compaction preserve raw shell output versus summarize it?
- Should rules be allowed to require confirmations (for example, “never run destructive commands without explicit approval”)?

## Acceptance Gates
- Host-scoped recall must not cross-apply stale facts to a different host without explicit operator confirmation.
- Auto-compaction must reduce context-limit failures in regression tests without hiding critical execution details.
- Retention and pruning controls must be test-covered and visible to operators.
- Rule precedence must be deterministic and documented.
- Product/operator docs must be updated to reflect real behavior before this epic is marked complete.
