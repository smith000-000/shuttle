# P5: Dual-Runtime Delegation And Escalation

## Purpose
Track the planned P5 mega-epic for explicit hybrid runtime behavior where one runtime remains authoritative for the session while Shuttle can delegate bounded subtasks to a secondary runtime.

## Core Position
- P5 is separate from P2.
- P2 continues the full outsource model with one authoritative runtime per session/task.
- P5 introduces explicit delegation, not silent fallback.
- Shuttle remains the sole owner of tmux interaction, shell observation, approvals, transcript mutation, patch application, and task/session persistence.

## Problem Statement
The current runtime model supports:
- `builtin` as a session-authoritative runtime
- `codex_sdk` as a session-authoritative runtime
- `codex_app_server` as a session-authoritative runtime

It does not support:
- builtin deciding a task is complex enough to escalate to Codex mid-task
- builtin deciding progress is stalled and outsourcing a bounded repair attempt
- Codex doing a bounded deep subtask while builtin remains the long-lived session owner

That hybridization needs its own design because otherwise runtime ownership, transcript continuity, plan state, and recovery semantics become ambiguous.

## Target Model
P5 should introduce:
1. One authoritative runtime per session/task.
2. An explicit Shuttle-owned delegation mechanism for bounded subtasks.
3. Structured delegate results that return to the authoritative runtime for reintegration.
4. Visible transcript/state records showing when delegation occurred, to which runtime, and what came back.

## Non-Goals
- No silent per-turn builtin fallback.
- No hidden runtime mixing inside a single reasoning turn.
- No direct runtime ownership of tmux, approvals, patch apply, or persistence.
- No generalized runtime marketplace in the first P5 slice.

## Open Design Areas
- escalation triggers:
  - complexity heuristics
  - repeated failure / stalled progress heuristics
  - user-explicit outsource requests
- delegate scope:
  - code-edit subtask
  - patch repair subtask
  - plan refinement subtask
  - research / deep analysis subtask
- reintegration:
  - message-only result
  - patch proposal result
  - plan update result
  - structured failure result
- transcript UX:
  - delegated-work event cards
  - runtime attribution
  - visibility of delegated prompts/results
- approvals and safety:
  - whether delegate proposals are always re-reviewed by the authoritative runtime
  - whether Shuttle should require explicit approval before certain delegate launches

## Proposed First Slice
- Keep `builtin` as the only initial authoritative runtime that can delegate.
- Allow delegation only to Codex-backed runtimes.
- Restrict delegate output to structured message / plan / proposal payloads.
- Feed delegate results back to builtin as explicit structured context rather than letting the delegate mutate Shuttle state directly.

## Acceptance Direction
- Delegation must be explicit in transcript/state.
- The authoritative runtime must remain the owner of the session/task.
- Shuttle must remain the only owner of shell state, approvals, execution, patch apply, transcript mutation, and persistence.
- Delegated failures must degrade cleanly back to the authoritative runtime without hidden reasoning loss.
